# 공유카드 이미지 생성 — 기술 스파이크 (BACKLOG S4)

디자인/브랜딩이 아직 확정되지 않아 최종 구현은 보류. 후보 기술과 트레이드오프만 정리.
(라이브러리 버전은 조사 시점 npm 최신 기준)

## 후보 1: 클라이언트 캡처 — `react-native-view-shot` (5.1.1) + `expo-sharing` (57.0.2, Expo SDK 번들)

- 방식: React Native로 카드 레이아웃을 화면 밖(또는 숨김 상태)에 렌더링 → `captureRef`로 PNG 캡처 → `expo-sharing`으로 네이티브 공유 시트 호출
- 장점: 서버 비용 없음, 오프라인 동작, 카드 레이아웃을 RN 컴포넌트로 짜므로 앱 UI와 스타일 재사용 쉬움
- 단점: 기기별 폰트/렌더링 미세 차이 가능, 캡처 타이밍/레이아웃 버그가 기기마다 다르게 날 수 있어 QA 부담
- Expo 호환: `expo-sharing`은 SDK 번들 패키지라 리스크 낮음. `react-native-view-shot`은 Expo config plugin 없이도 최신 버전에서 autolinking으로 동작하는 것으로 알려져 있으나, 실기기(M9)에서 실제 확인 필요

## 후보 2: 서버 사이드 생성 (Vercel Functions에서 이미지 렌더링, 예: `@vercel/og` 계열 satori 기반)

- 방식: `server/`에 카드 생성 엔드포인트 추가, listing 데이터로 서버에서 이미지 렌더링 후 URL 반환 → 클라이언트는 그 URL을 공유
- 장점: 모든 기기에서 픽셀 단위로 동일한 결과 보장, 워터마크/브랜딩 서버에서 일괄 관리·수정 쉬움 (바이럴 루프의 "워터마크로 자연 노출" 요구사항과 잘 맞음), 카카오톡 등 링크 공유 시 OG 이미지로도 재사용 가능
- 단점: 서버 호출 비용/지연 추가, `server/`가 Vercel Edge/Node 런타임 제약(폰트 임베딩 등) 고려 필요
- Expo 호환 이슈 없음 (클라이언트는 이미지 URL만 받음)

## 후보 3: `@shopify/react-native-skia` (2.6.9)

- 방식: Skia 캔버스로 직접 카드 렌더링·캡처
- 장점: 가장 유연한 커스텀 그래픽(그라디언트, 커스텀 폰트 렌더링 등)
- 단점: 셋업이 무겁고(네이티브 빌드 크기 증가) 이 프로젝트 규모 대비 과함
- 결론: MVP 단계에서는 후보 제외 권장

## 권장 (잠정, 최종 확정은 S4 디자인 확정 이후)

바이럴 루프상 "워터마크가 공유될 때마다 자연 노출"이 핵심 요구사항이라, 브랜딩 일관성이 중요한 **후보 2(서버 사이드 생성)** 를 우선 검토 권장. 다만 3주차 시점 서버 호출 비용/지연이 부담되면 **후보 1(클라이언트 캡처)** 로 시작해 MVP를 빠르게 내고, 트래픽이 늘면 후보 2로 전환하는 것도 합리적.

최종 라이브러리 선택과 카드 디자인(색상/레이아웃/워터마크 위치)은 사람 확인이 필요한 항목(BACKLOG S4)으로 남겨둠.
